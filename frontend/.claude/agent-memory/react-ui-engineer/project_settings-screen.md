The real Settings screen (Phase 1) replaced the `/settings` PlaceholderScreen. Feature dir `frontend/src/features/settings/` (SettingsScreen + api.ts + components/). Left SECTION RAIL (a NEW sub-nav pattern — the app's other consoles use horizontal Radix tabs) + stacked section cards, mirroring the SecurityTab (`features/team/components/SecurityTab.tsx`) form idiom.

**How to apply:** when extending Settings (Phase 2+), keep the seam + section-card + rail-focus pattern below; add new sections as `components/<Name>Section.tsx` wrapped in the shared `SectionCard` (its `<h2 id="section-<param>">` is the rail anchor).

- **Routes** (`app/router.tsx`): `/settings` → SettingsScreen; `/settings/$section` → same screen (reads the param via `useParams({ strict: false })`; valid sections organization/whatsapp/booking-rules/languages/team, unknown → show all). Two STATIC redirect routes out-rank the `$section` param route: `/settings/users` → `/team`, `/settings/commission` → `/care-partners` (the seeded nav children from `08_rbac_navigation.sql` point there). Lazy-loaded → own chunk.
- **Seam** (`GET/PATCH /api/v1/settings`): `getSettings`/`updateSettings` in `lib/backend/real.ts` + `lib/mock/settings.ts`, wired in `lib/backend/index.ts`, zod in `contracts.ts` (`SettingsSchema`, `UpdateSettingsRequestSchema`). Hooks `useSettings(enabled)` / `useUpdateSettings` in the feature api.ts. **PATCH carries NO Idempotency-Key** (config write, not money/booking — the controller documents it). Each supplied section is a FULL REPLACE (send the whole section object). PATCH `onSuccess` sets the returned DTO into the `['settings']` cache; both forms `reset`/re-seed from the returned section so dirty clears + server normalization reflects.
- **⚠️ WIRE QUIRK (VERIFIED live on :5054):** the settings DTO serializes the WhatsApp block as **`whatsApp`** (capital A) — .NET camelCases the C# `WhatsApp` property, lowercasing only the first char. The handoff contract wrote `whatsapp`; the running endpoint is authoritative. App-facing field is `whatsApp` (mock seeds it the same → no adapter). ALL other fields match the handoff exactly (facilityType/specialtyFocus are raw enum-ish STRINGS e.g. "hospital"/"multi_specialty", displayed read-only). The WhatsApp access TOKEN is never on the wire — kept out of the contract.
- **Permissions** (in-memory `can()`): settings query runs only when `tenant.settings.read`; without it the Organization/Booking-rules/WhatsApp sections collapse to ONE forbidden card (distinct from the GET-404 "no facility row" state — 404 detected via `error instanceof ApiError && error.status===404`). Editing gates `tenant.settings.update` (SecurityTab pattern: controls `disabled`, Save bar hidden). Languages + the More-settings links ALWAYS render (Languages available to everyone). Care Partners link card gates `commission.rules.read`.
- **Per-device language switcher** (Languages section): `setLanguage(lang)` exported from `app/i18n.ts` does i18n.changeLanguage + `localStorage['docslot.lang']` + `document.documentElement.lang`. i18n init now reads that key as the initial `lng` (was hardcoded 'en'). It is a DEVICE preference, distinct from the server per-user language (edited in Team & roles) — copy says "on this device".
- **Section rail focus:** navigating to `/settings/$section` focuses `#section-<param>` via `requestAnimationFrame` (so it wins over AppShell's route-change `main h1` focus, which runs later as the parent effect) + `scrollIntoView` honoring prefers-reduced-motion. Collapsed gated states carry `id="section-organization"` (sr-only h2) so gated rail links still land.
- i18n namespace `settings.*` (en+hi) in `app/i18n.ts`. Reused `team.security.unsaved` for the "unsaved changes" line (generic, bilingual). Related: [[live-api-seam]], [[screen-conventions]].
