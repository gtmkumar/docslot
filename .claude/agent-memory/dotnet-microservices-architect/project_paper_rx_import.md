Paper-prescription import feature (built 2026-07-02) on `docslot.patient_medical_history`: front desk transcribes a handwritten Rx into UNVERIFIED external records (photo attached), a clinician verifies later.

**Shape (all in the Clinical feature):**
- `source` column ('clinic' | 'paper_prescription' | 'patient_reported'); schema CHECK `chk_history_clinic_rows_verified` forces clinic rows to carry `verified_by_user_id` + `verified_at`, and `chk_history_verify_pair` makes verifier/timestamp both-or-neither.
- Endpoints on `ClinicalController`: `POST .../medical-history/import` (RequireAnyPermission create|intake, Idempotency-Key required via `IRequireIdempotency`), `POST .../{id}/verify` (update perm), `GET .../{id}/attachment` (read perm, consent+purpose+break-glass gated), and the list GET extended with source/externalDoctorName/verifiedAt/etc.
- `external_doctor_name` is field-encrypted (registry data_class `medical_history`); the scanned attachment is envelope-encrypted bytes stored via `IBlobStorage.PutAsync(kind="medical_history", batchId)`, mirroring `SetLabReportFileCommand`. All rows of one import share `import_batch_id` + the attachment pointer.

**Why:** clinic records are verified-at-creation by definition; imported ones are drafts. Drug-safety screening (`ListSafetyHistoryAsync`) deliberately includes UNVERIFIED external rows (fail-safe over-alerting) — do not "fix" that.

**How to apply / gotchas:**
- The EXISTING single create handler had to start stamping `source='clinic'` + verifier = caller/now or it violates the CHECK — done in `MedicalHistory.Create`.
- `MedicalHistory.FromRow` gained the new columns as OPTIONAL trailing params so `ListSafetyHistoryAsync`'s call site stayed untouched.
- Any test/seed that INSERTs a clinic-source `patient_medical_history` row directly must now supply `verified_by_user_id` + `verified_at` (fixed `PhiImpersonationRlsTests` seed).
- Idempotency footgun: don't set `Idempotency-Key` as a client DEFAULT header in tests — a later same-endpoint POST (e.g. second verify) replays the cached response. Send it per-request.

**Unified patient timeline (task #14, built 2026-07-02, REDESIGNED to fixed frontend contract):** `GET /api/v1/patients/{id}/timeline` (`GetPatientTimelineQuery` in `PatientTimelineFeatures.cs`) — merges non-draft prescriptions, lab reports, vaccinations, and paper-import document cards (grouped by `import_batch_id`) into one purpose-gated feed. Wire contract is FIXED by `frontend/src/lib/mock/contracts.ts` `PatientTimelineSchema` (match field names exactly — the frontend builds against it): `patient{patientSince,visitCount}`, `categories[{key,labelEn,labelHi,count}]`, `items[{itemId,category,occurredAt,title,subtitle,summary,tags,unverified,hasAttachment,ref{type,id}}]`. Endpoint: `RequireAnyPermission(prescription.read, report.read, medical_history.read)`. GATING MODEL (AUDITOR-VETOED the aggregate form; FINAL = PER-CATEGORY): each category is unlocked by (active consent) OR a patient-wide break-glass grant for THAT category's OWN resource_type (prescription→"prescription", lab_report→"lab_report", vaccination+document→"medical_history"). Break-glass grants are resource-type-scoped (ComplianceServices matches exactly), so a grant for one type must NEVER disclose another — an aggregate "any grant unlocks all" gate is a cross-type PHI-disclosure hole (auditor veto 2026-07-02). A category that is permission-open but consent/grant-closed is OMITTED (chip+items), same as the permission filter. Whole-request 403 only when NO readable category is accessible. Purpose-of-use: ONE entry PER accessed resource type (mirrors the sibling list reads), break-glass flag reflects only that type's grant (never a blanket `patient_timeline` entry misattributing one grant's justification). Permission filter still gates which categories exist at all (`IPermissionContext.Has`). Regression test: non-consented patient + "prescription"-only grant + caller with all 3 read perms ⇒ ONLY prescription chip/items, vaccination/lab hidden, and purpose log has break-glass=true ONLY for prescription. Strip is from `docslot.bookings` (MIN(booked_at) IST date + COUNT non-cancelled), NOT patient_tenant_links. `patientSince`/strip dates projected to Asia/Kolkata IN SQL (`(x AT TIME ZONE 'Asia/Kolkata')::date`) — do NOT `DateOnly.FromDateTime` a UTC timestamp (drops a day for early-IST). Summaries carry medication COUNTS only, never drug names (prescription: decrypt+`CountMedications`; document: count medication-type rows in batch). Titles decrypted in-policy (`PrescriptionFields.Diagnosis`/`.Medications`, `ClinicalFields.HistoryTitle`/`.HistoryExternalDoctorName`). ref.type: prescription/lab_report/medical_history(vaccination)/medical_history_batch(document). Test for "caller can't read a category": doctor-role user + a per-user DENY override in `platform.user_permission_overrides` on `docslot.report.read` (deny-wins) → lab_report omitted.

**Prescription-OCR proxy (task #13, built 2026-07-02):** `POST /api/v1/medical-history/extract-prescription` (`ExtractPrescriptionCommand` in `AiDocumentFeatures.cs`) — OCR a CALLER-SUPPLIED paper-Rx image into transcribable draft records for intake. Mirrors the lab-extract proxy STRUCTURE (`ISelfManagedTransaction` + `IDoNotCacheResponse`; `IAiOcrClient.ExtractPrescriptionAsync` on BOTH `HttpAiOcrClient` → `POST /ai/v1/extractions/prescription` and `StubAiOcrClient` → deterministic ONE medication record; provider chosen by `AiService:Provider` in DI, tests default to stub). KEY DIFFERENCES from lab extract: (1) NO consent gate — the caller supplies the image bytes (front desk holds the physical doc), so it processes a caller-supplied document not stored patient PHI, same reason the paper-Rx IMPORT skips consent; keeps only the `IsLinkedToTenantAsync` 404 guard. (2) rawText IS surfaced (lab omits it) — the desk verifies the transcription. Gate: `RequireAnyPermission(docslot.medical_history.create, docslot.medical_history.intake)` (both is_dangerous → tenant_admin holds NEITHER, so tenant_admin is the 403 test principal). X-Purpose-Of-Use required (422) + forwarded to AI. Response DTO `PrescriptionExtractionDto` (available/extractionId/overallConfidence/externalDoctorName/recordedDate/records[{recordType,title,description,confidence}]/rawText/source) — added `source` beyond the team-lead's listed fields (house AI-DTO provenance convention; frontend schemas are non-strict). No-cache test trick: same Idempotency-Key + two different patients → distinct `stub-rx-{patientId:N}` extractionIds proves no replay. Image cap 28M base64 (same as import), contentType must be image/*.

**LIVE-DB RBAC SWEEP GAP (was a blocker; RESOLVED 2026-07-02):** adding a permission to the registry does NOT auto-grant it to super_admin on the already-built `docslot_platform` DB — the universal sweep lives at the end of `database/10_roles_grants.sql` (idempotent) and must be re-run. After the `docslot.medical_history.intake` key was added, super_admin held 134/135 and `RbacNavigationTests.SuperAdmin_Resolves_Full_Registry` failed. Fix (needs user/privileged approval — auto-mode classifier blocks agent RBAC writes on shared infra): `psql -d docslot_platform -f database/10_roles_grants.sql` (or the `INSERT ... super_admin ... ON CONFLICT DO NOTHING` sweep). The sweep was applied by a privileged actor by the next slice, so `RbacNavigationTests` went green again (135/135). LESSON: after any product SQL adds a permission to the live DB, someone with privileges must re-run the 10_roles_grants super_admin sweep — the agent cannot. See [[project_slice08_rbac_navigation]].
