Phase A of docs/PRESCRIPTION_CONSULTATION_PLAN.md — the consultation composer backend (endpoints `POST /api/v1/consultations`, `PATCH /api/v1/consultations/{id}`, `POST /api/v1/consultations/{id}/finalize`) built on `docslot.prescriptions` as a draft→finalized state machine. All three gated on `docslot.prescription.create` in Phase A (Phase B relaxes create/patch to `docslot.prescription.draft`). Lives alongside the clinical vertical (ConsultationFeatures.cs, ClinicalController).

**Why:** one consultation record, two roles (intake vs doctor), one signing transition; the finalize author (doctor) is server-derived from `docslot.doctors.user_id`, never client-asserted (MCI).

**How to apply:**
- **Signer CHECK regression:** `docslot.prescriptions` now has `chk_prescriptions_signed_rows_have_signer` — any row with status IN ('finalized','delivered','amended') MUST set `finalized_by_user_id` + `finalized_at`. `Prescription.Issue`/`Amend` set the signer = authenticated caller. Any DIRECT SQL seed of a finalized prescription (tests, migrations) MUST include both columns or it fails. Partial unique index `uq_prescriptions_booking_draft` = one draft per booking.
- **vitals + investigations are UNENCRYPTED standard PHI** (JSONB, purpose-of-use gated on read, deliberately NOT in encrypted_fields_registry). chief_complaints/examination/diagnosis/medications stay field-encrypted like [[slice03b-clinical-phi]]. Repo reads vitals/investigations via `::text`, medications via `#>> '{}'`.
- **Idempotent drug screening on finalize:** only run `IDrugSafetyScreeningService.ScreenPrescriptionAsync` when zero alerts already exist for the prescription (a prior blocked finalize already generated them) — prevents duplicate alert rows on the override retry. Unoverridden high/critical alerts block signing unless `overrideReason` given; override marks them overridden (never deletes).
- **medicationsJson has TWO shapes — screening MUST parse both (clinical-safety):** legacy `[{name,dose,frequency,duration}]` (all strings) AND the composer STRUCTURED shape `[{name,strength,form,dose:{morning,noon,night},timing,durationDays,instructions}]` where `dose` is an OBJECT. DrugSafetyScreeningService.ParseMedications is JsonDocument-based + PER-ITEM tolerant (only `name` matters to the rules). Prior bug: a typed deserialize (dose:string) threw on the object shape → defensive catch returned [] → the WHOLE screen silently no-oped → a penicillin-class drug finalized against a severe penicillin allergy with zero alerts (live PRX-2026-07-02203). Regression test uses the structured shape (ClinicalPhiTests.Consultation_Finalize_Screens_STRUCTURED_Medications_Shape_Against_Allergy) — the legacy-shaped allergy-block test did NOT catch it.
- **UpdateDraftAsync uses COALESCE per field** (null arg = leave unchanged, partial autosave never wipes). GOTCHA: the COALESCE params need explicit `::text`/`::int`/`::jsonb` casts or Npgsql sends untyped params → PG "could not determine data type of parameter $N" (500). Same class of gotcha as [[slice08b-read-endpoints]] untyped-NULL casts.
- **Doctor test user:** ClinicalWebAppFactory seeds `DoctorUserId` (platform user, tenant_owner) LINKED to the fixture doctor row via `docslot.doctors.user_id`; the admin (AdminUserId) is deliberately NOT doctor-linked → it's the "non-doctor with prescription.create → 403" case. `tenant_owner` holds prescription.create/read (all docslot tenant/self perms); the seeded `doctor` ROLE does NOT include prescription perms.
- **Phase D §6 booking self-scoping (doctor queue):** GET /api/v1/bookings (list) + /bookings/{id} (detail) now use a NEW `[RequireAnyPermission("docslot.booking.read","docslot.booking.read_self")]` attribute (mediq.Api/Authorization/RequirePermission.cs; `anyperm:` policy prefix + AnyPermissionAuthorizationHandler, registered in ApiServiceExtensions) — the OR gate (stacking [RequirePermission] is AND). ListBookingsQueryHandler/GetBookingQueryHandler resolve scope via `BookingReadScope.ResolveDoctorFilterAsync` (DocslotQueries.cs): tenant-wide `booking.read` = reception unchanged; `booking.read_self` only = confined to `GetDoctorByUserIdAsync(ctx.UserId)` doctor row (forces filter.DoctorId, ignoring any client ?doctorId — no widening; detail 404s another doctor's booking, no existence leak). The 'doctor' role holds booking.read_self but NOT booking.read. IPermissionContext is resolved for EVERY authenticated request (PermissionResolutionMiddleware, not gated on [RequirePermission]) so handlers can read it safely. NOT done (excluded): patients-list + overview scoping.
- **Nav failure was transient:** AuthRbacFlowTests…SuperAdmin (bookings nav menu had no children — stale live-DB seed vs uncommitted 08_rbac_navigation.sql) resolved once the nav menus were re-seeded; full suite now 373/373.
