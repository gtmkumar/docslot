## Wave 4 (Remediation) â€” gotchas to remember
- **Card `tone` prop, NOT appended bg classes.** `Card` (components/ui/Card.tsx) owns bg/border/text via `tone='surface'|'emphasis'`. The Live Queue dark card uses `tone="emphasis"`. Appending `bg-ink` to a Card that hardcodes `bg-surface` is a Tailwind-v4 source-order conflict (bg-surface wins â†’ invisible card). If you need a dark card, use the tone, don't append a bg.
- **SlideOverHost URLâ†”store hydration order** (don't reintroduce the refresh bug). The storeâ†’URL writer no-ops on its FIRST invocation (`writerFirstRun` ref) and otherwise only writes when the panel TYPE differs from `lastSyncedType`. This lets a deep-linked `?panel=&id=` survive a hard refresh â€” the writer must never clear a param while the store is still catching up to the URL on mount.
- **Theme/density UI**: `components/layout/AppearanceMenu.tsx` (disclosure popover, outside-click+Esc close) wired to `useUI` setTheme/setDensity, mounted in the Topbar. AppShell applies `data-theme`/`data-density` to `<html>`.
- **Responsive shell**: AppShell renders a static sidebar rail â‰Ąmd and a mobile drawer <md. The drawer is `components/layout/MobileNavDrawer.tsx` â€” a **Radix Dialog** (focus-trap + Esc + overlay close for free; the old hand-rolled `<div role=dialog>` had none) reusing the SAME `<Sidebar>` (no nav duplication), slide-in via `slideInLeft`. Topbar hamburger (`onOpenMenu`) shown <md. `Sidebar` fills its container (`h-full w-full`) + optional `onNavigate` closes the drawer on route change.
- **Topbar actions overflow**: `components/layout/TopbarActions.tsx` â€” roster + book-time render inline at lg+, collapse into a "more" disclosure menu (outside-click+Esc) below lg; "+ New walk-in" is icon-only below sm. AppearanceMenu stays a fixed control in Topbar. All still `can()`-gated. i18n `topbar.moreActions`.
- **Tabs reflow**: Team/Developers tab triggers use `shrink-0 whitespace-nowrap`; the Developers TabsList is `min-w-0 flex-1 overflow-x-auto` so 4 tabs scroll horizontally rather than push the action button off-screen at 360px.
- **404**: `notFoundComponent: NotFoundScreen` on BOTH the root route (fallback) AND the pathless `authed` layout â€” the authed one renders the on-brand 404 INSIDE AppShell (sidebar/topbar usable) for signed-in users; unauthenticated users still bounce to /login via the layout beforeLoad. NotFoundScreen uses EmptyState + a primary "Back to Overview" action. i18n `notFound.*`.
- **a11y dialog titles (D7, swept)**: every Radix Dialog has a `Dialog.Title`. SlideOver + MobileNavDrawer have real/visually-hidden titles; cmdk's `Command.Dialog` gets a `VisuallyHidden` Radix `Dialog.Title`+`Description` (distinct `command.title`/`command.description`). SlideOver has an optional `description` prop (visually-hidden `Dialog.Description`); omitted â†’ `aria-describedby={undefined}` (Radix warning-free opt-out). AppearanceMenu/TopbarActions are `role="menu"` popovers, NOT dialogs.
- **D6 focus trap + RETURN (non-obvious gotcha)**: Radix traps Tab by default (nothing breaks it). BUT slide-overs/drawer open PROGRAMMATICALLY from the UI store (controlled `open`), not via a Radix `Dialog.Trigger`, so Radix had no trigger to restore focus to on close â†’ focus fell to `<body>`. Fix in SlideOver + MobileNavDrawer: `useEffect` captures `document.activeElement` when `open` flips true; `onCloseAutoFocus` (preventDefault + trigger.focus()) restores it. ANY new store-opened Radix dialog needs this pattern.
- **Code-splitting (DONE â€” no more single-chunk warning).** Route screens (Overview/Team/Developers) are `React.lazy` in `app/router.tsx`; Suspense boundary + `RouteFallback` skeleton live in AppShell's `<main>`. ALL slide-over panel bodies are `React.lazy` in `SlideOverHost` (wrapped in `<Suspense fallback={null}>` â€” `renderPanel(panel, closePanel)` helper; URLâ†”store sync logic untouched). Shell + LoginScreen stay in the entry chunk. `vite.config.ts` `build.rollupOptions.output.manualChunks` splits vendor libs (tanstack/radix/i18n/forms/icons) â€” do NOT add a `react`/`react-dom` manualChunk (produces an empty chunk; the compiler runtime keeps them in the entry chunk). Result: largest chunk ~365kB (was ~852kB single), no >500kB warning.
