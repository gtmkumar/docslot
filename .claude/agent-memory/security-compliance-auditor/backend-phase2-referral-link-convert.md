Phase-2 organic attribution path #3: PUBLIC REFERRAL LINK click→convert. Broker shares `/api/v1/ref/{shortCode}` → click logged (IP SHA-256 hashed, UA trimmed 50, httponly/secure/lax session cookie) → 302 to OUR server-built `wa.me/{tenant primary_phone}?text=...(Ref: BRK-xxxxxxxx)` → patient's prefilled WhatsApp message carries the code → inbound handler regex-detects `BRK-[0-9A-Fa-f]{8}`, stashes referralLinkId/BrokerId on a NEW conversation, and at booking CONFIRM mints an `auto_verified` `referral_link` attribution via the SAME CreateAttributionCommand + MarkConvertedAsync. **THE ONE CONDITION — best-effort tx-poisoning (gate 5), MEDIUM:**
Nested dispatches REUSE the ambient EF transaction (UnitOfWork.cs:44 `existing ?? BeginTransactionAsync`; nested UnitOfWorkBehavior.CommitAsync is a NO-OP when ownsTransaction=false). So booking+attribution+click-conversion COMMIT together at the OUTERMOST UnitOfWorkBehavior. The handler's try/catch around the attribution (ProcessInbound...Handler.cs:340-358) catches the .NET exception, BUT if any nested statement raised a **PostgresException** (DB-level error), Postgres aborts the WHOLE tx (25P02) — the C# catch can't un-abort it, and the outer SaveChangesAsync/CommitAsync then fails/rolls back → **the patient's booking silently vanishes** (amplified: WhatsAppController.cs:113-117 swallows the commit failure and returns 200 to Meta → no retry, patient already got a "confirmed" outbound). FOR THE DESIGNED FAILURE MODES the guarantee HOLDS — they're all pure-C# throws (ExistsAsync dup→BusinessRuleException; CanEarn/IsLinked→ForbiddenException; booking-not-found→KeyNotFoundException; self-referral→score-only no-throw; 23514→can't fire). Residual = an UNANTICIPATED PostgresException (future CHECK on attribution_source/fraud_flags, 23505 unique race, transient serialization/conn blip) poisons the shared tx. CONDITION: make the referral attribution genuinely isolated — run it in a SEPARATE transaction/connection (commit the booking FIRST, then attribute best-effort out-of-band), OR wrap the nested dispatch in a SAVEPOINT so a DB error rolls back only to the savepoint and the booking tx survives. Until then "best-effort" is only true for the enumerated C# failures, not for arbitrary DB errors.

**LOW findings (non-blocking):** (1) unsalted SHA-256 IP hash → keyed HMAC. (2) Public anon /ref/{code} is unauthenticated + tenant-less + unbounded-write (1 click row + click_count bump per hit) — enumeration/flood DoS surface; YARP gateway rate-limit (intended trust boundary) not yet built → add per-IP rate limit at the edge before prod. (3) Test gap: NO test drives a FAILING referral attribution that then asserts the booking still commits — the best-effort claim is asserted by comments, not proven. Recommend a fault-injection test (e.g. pre-existing attribution / DB error) asserting booking persists.
