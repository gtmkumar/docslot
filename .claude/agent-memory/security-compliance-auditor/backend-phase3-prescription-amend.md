## Why clean PASS (all gates verified source + live + test)
- **PHI-at-rest:** all 4 registered clinical fields (diagnosis/medications/examination/chief_complaints — all in encrypted_fields_registry, data_class=medical_history legal_basis=consent, live-verified) encrypted via IFieldEncryptionService with the SAME PrescriptionFields refs + EncryptionContext("prescription", patientId) as Issue. Handler encrypts BEFORE Prescription.Amend(); meds always encrypted (NOT NULL), other 3 EncOrNull. Test asserts amendment.diagnosis ciphertext-at-rest. amendment_reason is operator JUSTIFICATION free-text, NOT clinical PHI, NOT in registry — acceptable unencrypted (it's the "why", not patient content); it is NOT echoed into audit ChangeSummary or the integration event.
- **Atomicity / non-overwrite:** AmendPrescriptionCommand is NOT ISelfManagedTransaction ⇒ runs under UnitOfWorkBehavior = ONE tenant-scoped tx (SET LOCAL app.tenant_id → handler → SaveChanges → Commit). Load → MarkPrescriptionSupersededAsync (conditional UPDATE) → encrypt → AddPrescriptionAsync (INSERT) → audit all in that tx. Mark-FIRST single-winner: `UPDATE ... SET status='amended' WHERE id=@ AND tenant_id=@ AND status IN ('finalized','delivered') RETURNING status` → 0 rows ⇒ false ⇒ 409. Concurrent double-amend: both pass the racy app pre-check, only one wins the UPDATE; loser 409s. Original can NEVER be left superseded-with-no-successor (UPDATE+INSERT same committed tx; later failure rolls flip back).
- **State machine:** prescriptions.status DB CHECK IN ('draft','finalized','delivered','amended') (live). Amendable={finalized,delivered}. draft→422 (app) + UPDATE no-match; already-`amended`→422 (load sees it first) + UPDATE no-match (double-guarded). Test proves re-amend of superseded original → 422, ghost id → 404.
- **RLS / tenant:** new cols on existing RLS table (policy tenant_isolation_prescriptions FOR ALL, USING tenant_id=current_tenant_id() OR current_impersonated_tenant(); FOR ALL w/ null WITH CHECK ⇒ USING reused ⇒ INSERT/UPDATE constrained). docslot_app NOBYPASSRLS, grants INSERT/SELECT/UPDATE only (no DELETE). Load + conditional UPDATE both pinned WHERE tenant_id=@. GetPrescriptionAsync now sets cmd.Transaction = CurrentTransaction (line 60) so the raw reader runs under the amend tx's GUC (closes the prior medical-history INFO nit, now applied to BOTH presc + medhist reads). tenant_id from RequireTenant()=JWT TenantClaim (no header fallback). FK supersedes_prescription_id is NO ACTION (confdeltype=a) — fine, app proves same-tenant lineage (original loaded WHERE tenant_id=@); no DB tenant-match constraint but unreachable cross-tenant via app.
- **SoD / permission:** NEW `docslot.prescription.amend` is_dangerous=t scope=tenant (live), DISTINCT from prescription.create (issue≠amend). Auto-granted to tenant_owner (03 sweep: all docslot tenant/self perms) + super_admin (10 universal sweep). Live: ONLY super_admin+tenant_owner hold amend (create additionally on qa_*/smoke_* test-artifact roles — pre-existing test seeding, not the amend perm). `doctor` role holds NEITHER create/read/amend by design (doctor gets medical_history.* + self-scoped only; prescriptions go through owner/admin/staff) — NO parity gap.
- **Audit:** writer/hash-chain UNCHANGED (no audit/crypto/consent file touched; verify_audit_chain()=0 live). amend audit: Action="amend", ResourceLabel=PRX number, ChangeSummary references superseded NUMBER/ID only "(...; PHI encrypted)", Purpose=treatment LegalBasis=consent — NO clinical content. Integration event docslot.prescription.amended carries prescription_id/supersedes_prescription_id/patient_id/booking_id — IDs ONLY.
- **No consent gate on amend WRITE:** intentional + consistent (Issue + medhist writes not consent-gated; the READ is the gated disclosure — read path stays consent+purpose+break-glass gated, untouched).

## Follow-ups (INFO only, non-blocking, not conditions)
- No forward pointer: reading the original does NOT surface its amendment id (only amendment→original via SupersedesPrescriptionId). A reader of the superseded original sees status='amended' but must list to find the successor. UX/safety nicety, not a control gap (status flag already signals "superseded").
- No DB constraint forcing amendment.tenant_id == superseded.tenant_id, nor blocking self/cyclic supersedes chains. App-enforced + unreachable via the endpoint; could add a trigger/CHECK as defense-in-depth if direct-SQL writers ever appear.
