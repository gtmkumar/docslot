## Backend change (PHI read-shape) — non-PHI scalar enrichment, ACCEPTABLE
`MedicalHistoryDto` gained `Severity, Icd10Code, StartedDate, EndedDate`. Verified these are PLAIN columns (live `information_schema`: severity/icd10_code = varchar, started_date/ended_date = date) and NOT in `encrypted_fields_registry` (live: only `.title` + `.description` registered, data_class=medical_history legal_basis=consent). So no new encrypted-field obligation. Read path (`ListMedicalHistoryAsync`/`GetMedicalHistoryAsync`) SELECTs them under the SAME consent+purpose+RLS gate that already decrypts title/description — no new exposure class, gate UNCHANGED. Fixes a real data-loss bug: the edit form's PUT treats missing as null, so severity/icd10/dates were silently wiped on every UPDATE because the read never returned them. Test now asserts severity round-trips + survives an edit. DTO is JSON name-keyed (not positional) ⇒ no field-order mismatch risk across C#/zod even though both list the 11 fields in the same order. See [[backend-phase3-medical-history-crud]] (the write path this complements).

## Frontend PHI surface — all controls intact
- **X-Purpose-Of-Use is an HTTP HEADER, not a URL param.** `apiFetch` sets `headers['X-Purpose-Of-Use'] = req.purposeOfUse` (api-client.ts:135); Idempotency-Key also a header (:136). The 3 previously-unparameterized list hooks (usePrescriptions/useLabReports/useAbdmRecords) now thread `purpose` and are `enabled` only once purpose exists ⇒ no purpose-free clinical read reaches the wire. `getPatientConsent` is the ONLY purpose-free clinical GET (correct).
- **PHI never to URL/log/storage.** The 3 new panel types (`createHistory`,`editHistory`,`clinicalBreakGlass`) are in `TRANSIENT_SET` (SlideOverHost.tsx) ⇒ `isUrlPanel`=false ⇒ `panelToSearch` returns `{}` (clears any param). `editHistory` carries decrypted title/description in `entry` and is correctly transient; `searchToPanel` has NO restore case + they're not in PAYLOADLESS ⇒ a refresh/deep-link cannot reconstruct them (re-entry must re-declare purpose). grep: ZERO `console.*`/`localStorage`/`sessionStorage` under `src/features/patients/`. PHI lives only in form-local React state + the POST/PUT body + the RQ cache's normal lifetime.
- **Break-glass UI only SURFACES the affordance; every gate is server-side.** `ConsentBlocked` gates the button on `usePermissions().can('docslot.medical_access.break_glass')` (NOT role-in-JSX; grep confirms zero `role===` in the clinical surface). The panel only POSTs `{patientId,resourceType,resourceId,justification}` to `/security/break-glass`; the actual override authorization is the grant table + read handlers (audited PR#22 → [[backend-phase3-breakglass-unlock]]). `resourceType` is the restricted zod enum `['prescription','lab_report','medical_history']` — **ABDM excluded** (matches the DB CHECK + BreakGlassValidator). Justification client-validated ≥10 (server re-validates). On success invalidates `['clinical']` namespace ⇒ gated read re-runs.
- **ABDM is NOT break-glassable (preserved on the client too).** AbdmDetailPanel imports no ConsentBlocked/break-glass; on consent failure it shows a locked EmptyState. AbdmTab wires NO break-glass.
- **Bilingual** en+hi present for all new keys (consentBlocked/breakGlass/history.*). Slide-over modality, design tokens, Idempotency-Key on POSTs all honored.

## INFO (non-blocking, cosmetic)
AbdmDetailPanel detects consent-block via `error instanceof ConsentRequiredError` (mock only). In REAL mode an ABDM consent denial is a 403 `ApiError`, which falls through to the GENERIC error EmptyState instead of the dedicated "no consent" copy. Fail-SAFE (no data leak, no bypass affordance — ABDM intentionally has no break-glass), purely a UX-copy gap. The clinical (non-ABDM) panels use `isConsentDenied()` which DOES handle both ConsentRequiredError AND ApiError(403), so they're correct. If ABDM detail ever wants the dedicated copy in real mode, broaden its check to `isConsentDenied(error)` (but keep NO break-glass affordance for ABDM).
