**HIGH (the load-bearing CONDITION) — link reuses a NON-DANGEROUS permission for a national-network PHI disclosure.** Controller gates link on `docslot.abdm.records.create`, which in the product RBAC catalog (03_docslot.sql:858) is **is_dangerous=false**. The role seed grants tenant_admin EVERY docslot tenant/self perm WHERE is_dangerous=false (03:880), so **tenant_admin (and tenant_owner) auto-hold `.create`** ⇒ any tenant_admin can publish a patient's care context to the national ABDM network. Push (`.create`) only stores LOCALLY (encrypted) — link is the FIRST action that actually egresses PHI to the national exchange, a materially higher privilege. The danger-flag is exactly the fence that excludes a perm from the broad tenant_admin grant; reusing a non-dangerous perm under-classifies a national-network disclosure. FIX (maintainer offered): add a DISTINCT `docslot.abdm.records.link` permission, scope=tenant, **is_dangerous=true** (schema + bundle regen + grants + the `[RequirePermission]`), so link requires an explicit dangerous grant distinct from local-create — same spirit as approve≠execute on payouts and amend≠create on prescriptions. (Alternatively, if maintainer keeps `.create`, at minimum flip `.create` to is_dangerous=true — but a distinct key is cleaner and preserves local-push ergonomics.)

**MEDIUM — P3 ignores MarkAbdmRecordLinkedAsync's bool return.** On a concurrent double-link the 2nd UPDATE returns false (0 rows, already linked) but the handler unconditionally writes a 2nd audit "link" row + publishes a 2nd `record.linked` event, and (with a real nha adapter) P2 would have called the network twice. No money, no wallet, no PHI leak, row stays single-linked ⇒ bounded, NOT a blocker (cf. the ExecutePayout HIGH which DID double-credit). FIX: early-return success when the conditional UPDATE returns false (mirror the ExecutePayout rows-affected guidance) so audit/event fire exactly once per real linkage.

**INFO — DisabledAbdmGateway decline maps to 422.** A `none`/not-configured environment returns BusinessRuleException→422; semantically a 503/501 is more accurate for "integration not wired", but it's honest (record stays unlinked, no fake success). Cosmetic.
