**HIGH (CONDITION for clearance before a REAL payout gateway is wired) — ExecutePayoutCommand not idempotent / not concurrency-safe:**
ExecutePayoutCommand does NOT implement IRequireIdempotency (unlike Create/Reschedule/BookingAction). Isolation = read committed. Two concurrent executes on one approved payout: both GetByIdAsync read 'approved' → both pass CanExecute → BOTH call gateway.SendAsync (double disbursement in a live adapter). R1 MarkPaidAsync (WHERE status='approved') succeeds+locks; R2's same UPDATE blocks then re-evaluates post-commit → 0 rows (payout row stays single-'paid', good). BUT the handler IGNORES MarkPaidAsync's affected-row count and unconditionally proceeds: ReadyToPayAttributionIdsAsync (now empty) + **ApplyPaidAsync(gross) debits ready_to_pay again (GREATEST floors, no negative) and DOUBLE-CREDITS lifetime_paid_inr**. Bounded (no cross-tenant leak, no negative bucket, payout not falsely paid) → HIGH not BLOCKER, and stub gateway is harmless today. FIX: mark ExecutePayoutCommand IRequireIdempotency AND/OR check MarkPaidAsync rows-affected==1 before wallet/attr moves (early-return if 0), ideally SELECT ... FOR UPDATE the payout row at handler entry. Same applies to ApprovePayout (lower stakes).

**MEDIUM (note, not blocking):** ExecutePayout calls gateway.SendAsync INSIDE the open UoW transaction (UnitOfWorkBehavior) — a real network adapter holds a DB tx open across a network round-trip (lock duration / pool starvation). Consider executing the gateway call outside the tenant-scope tx (record intent, call, then a short tx to finalize).

**LOW:** Unverified post-hoc claim that never clears (patient_denied/no_response) leaves money in pending_inr forever (phantom pending) — display-only, never payable (only ready_to_pay pays). No verification-denial→pending-reversal path. Cosmetic.
