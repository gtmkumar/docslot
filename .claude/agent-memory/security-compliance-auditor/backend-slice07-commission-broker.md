**BLOCKER 1 — Broker self-service IDOR / cross-broker financial leak (gate 5):** `me/wallet`, `me/links` (GET+POST) take `[FromQuery] Guid brokerId` (`CommissionController.cs:61,67,73`) gated only by `commission.broker.read_self`. NO check that brokerId belongs to the caller; NO broker_id JWT claim exists to bind "self"; wallet query is `WHERE broker_id=@p0` only (`CommissionRepositories.cs:204`), no tenant/self scope. Any holder of read_self can pass ANY brokerId → read any broker's wallet/links and CREATE referral links under another broker's identity. DPDP unauthorized access + financial confidentiality. No test covers cross-broker confinement. FIX: derive the caller's broker_id server-side (add a broker_id claim at login for broker users, or look up broker by the caller's user/phone) and ignore/validate the query param against it; reject mismatch 403.

**BLOCKER 2 — Pool-safety cross-tenant GUC bleed (gate 8, carried from 03b, now manifesting):** `UnitOfWork.SetTenantScopeAsync` uses session-scoped `set_config('app.tenant_id', …, false)` (is_local=false) on a manually-pinned connection (`UnitOfWork.cs:22-28`). The comment claims a "defensive reset at request scope" that DOES NOT EXIST in code; no reset middleware; connection string doesn't configure reset. The Dashboard_Summary parallel-test flake is the canary: app.tenant_id from request A persists on a pooled connection reused by request B. Under prod concurrency this means RLS-scoped clinical reads can run against the WRONG tenant's GUC → cross-tenant PHI exposure. NOT just a test flake. FIX (priority HIGH, before any concurrent prod traffic): wrap each request's DB work in an explicit transaction and use `SET LOCAL` (tx-scoped, auto-cleared on commit/rollback) — the cleanest; OR set+RESET the session GUC in a request-scoped middleware (set on entry, `set_config('app.tenant_id','',false)` in a finally); OR a dedicated per-request connection. The set-without-reset on a session GUC + pooling is the root cause.
