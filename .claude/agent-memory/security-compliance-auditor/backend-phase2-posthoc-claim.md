**MEDIUM (CONDITION for clearance) — one-directional single-live-OTP guard:**
The inbound handler (ProcessInboundWhatsAppMessageCommandHandler:68-90) checks CONSENT first, then claim (4b), and its comment RELIES on the invariant "a patient never has BOTH pending at once, so this ordering is unambiguous." But the guard is only enforced on ONE side: PostHocClaimService.SendForClaimAsync:42-43 throws if a pending CONSENT exists before minting a claim. PatientConsentService.SendForBehalfBookingAsync:30-55 does NOT check for a pending CLAIM OTP — it only supersedes prior pending *consent*. So: broker files a claim (24h TTL), then within that window a behalf-booking names the same patient → patient now has BOTH pending. Patient's reply is consumed by consent (checked first); if the reply was the CLAIM code it's misrouted to consent as a wrong-code attempt → claim never confirms, lapses to no_response (reversed). Money-SAFE (no phantom credit, no cross-tenant leak) → MEDIUM not BLOCKER, but it VIOLATES the documented invariant the handler leans on. FIX: add the symmetric guard in SendForBehalfBookingAsync — reject (or supersede) when a pending claim OTP exists for (tenant, patientPhone). Mirror the cross-table check that SendForClaimAsync already does.

**LOW (newly material, pre-existing) — reversed claims still consume monthly cap:**
BrokerEarnedThisMonthAsync (AttributionRepository:83) sums `commission_amount_inr WHERE commission_status <> 'rejected'`. A reversed/no_response claim ends 'reversed' (NOT 'rejected'), so it STILL counts toward the broker's monthly cap base → shrinks `remaining` headroom (CommissionCalculator.Calculate:137-141) for later attributions. Direction is CONSERVATIVE-AGAINST-BROKER (can only lower future commission, never overpay/inflate) → not a fraud/platform-loss risk. Pre-existing filter (slice 07), but post-hoc reversals make it more common. Suggested: change filter to exclude 'reversed' (and 'rejected') from the cap base, or sum only earned/ready_to_pay/paid.
