Only the secondary hash-chained audit_log (write_audit_best_effort) is best-effort (pre-existing). Fix makes the AI-side gates actually fire in HTTP mode (improves posture; the triage purpose-gate was being masked by the 404 before).

**THE CONDITION (MEDIUM, F1 — sourceUrl local-file-read, NOT SSRF):** .NET forwards client-supplied `sourceUrl` raw to AI extract. AI treats it as a LOCAL FILESYSTEM PATH: `os.path.exists(source_url)` + `os.path.getsize` + `ocr.extract_lab_report(source_url)` (extractions.py:89-100) — it does NOT fetch a URL over the network, so the real class is arbitrary local-file read / path traversal on the AI host, not network SSRF. slice-11 is the NEW edge exposure of this primitive. Mitigants keeping it non-blocking: (a) stub is the dev/test DEFAULT and StubAiOcrClient ignores sourceUrl entirely → zero exposure in default config; (b) HTTP mode is explicitly pre-prod/unverified; (c) file content is NOT returned to the .NET caller (rawTextPreview omitted); (d) gated behind is_dangerous report.read + consent + tenant-link. REMEDIATION (preferred, trivial): .NET drops/constrains sourceUrl to an opaque blob handle the AI resolves against a controlled store — do this BEFORE AiService:Provider=http is enabled anywhere reachable by real users. A URL allow-list alone is insufficient (it's filesystem paths, not URLs) — must reject absolute/relative paths.
