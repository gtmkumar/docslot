**BLOCKER (ORIGINAL, now fixed) — ungated caller-supplied `sourceUrl` = authenticated arbitrary local-path file/OCR read (ai_service/app/routers/extractions.py).** Both endpoints accept `sourceUrl: str` (schemas.py:88/126, no config gate) and treat it as a local filesystem path: lab lines 96-103, prescription lines 251-258, then `ocr.py:33 Image.open(image_path)`. The endpoints REQUIRE a user JWT (prescription refuses token_use=="service" at line 204; lab's enforce_phi_gate refuses service too) — i.e. they are designed to be reached with the SAME user token staff hold. A staff caller reaching the AI service directly can send `{relatedPatientId:<their linked patient>, sourceUrl:<any path>}`: the file-read is NOT tenant-scoped (orthogonal to the patient gate), so any image readable by the AI process is OCR'd and its transcription returned as `rawText` + persisted — cross-tenant PHI (other tenants' scan blobs on the host), plus an arbitrary file existence/size oracle (os.path.exists→400, os.path.getsize). The .NET client NEVER forwards sourceUrl (AiOcrClients.cs:27-29 comment calls it "an arbitrary-file-read primitive"; prescription forwards imageBase64 only) — but the AI service is self-defending (own consent gate/service-wall/tenant-from-JWT), so it must not expose it either. Prior audit [[backend-phase4-ai-ocr-rag-seam]] flagged the LAB sourceUrl as MEDIUM "constrain before HTTP mode"; HTTP mode is now the live/verified config and the constraint was never added, and this PR extends the primitive to the prescription endpoint.

Required fix: (1) prescription endpoint — remove the `elif body.sourceUrl:` branch (only imageBase64 is ever used in prod); (2) lab endpoint — dev-gate sourceUrl behind an explicit non-prod settings flag (default False), else reject/ignore and use only the server-generated sample or a SERVER-resolved blob handle; (3) test: user JWT + sourceUrl=arbitrary path with flag off → rejected (not read), both endpoints.
