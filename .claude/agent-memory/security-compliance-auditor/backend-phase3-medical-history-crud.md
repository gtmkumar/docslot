NEVER the encrypted title/description. Category-only metadata in a tenant-scoped hash-chained audit is acceptable (indicates a class of record exists, not its content). NO integration event published on create/update (grep clean) — no RabbitMQ/outbox PHI-leak vector. AuditTrailWriter seam UNCHANGED (append-only hash-chain intact).
- **Tenant isolation:** INSERT runs in the command's UoW tenant-scoped tx (UnitOfWorkBehavior → BeginTenantScopeAsync SET LOCAL app.tenant_id); RLS policy `tenant_isolation_medical_history` is FOR ALL USING-only (USING reused as WITH CHECK) ⇒ cross-tenant INSERT/UPDATE rejected. UPDATE/GET pinned WHERE history_id+tenant_id AND under RLS (docslot_app NOBYPASSRLS). tenant_id comes from `RequireTenant()`=currentUser.TenantId = signed JWT TenantClaim (RequestContext.cs:43, NO X-Tenant-Id header fallback for authed callers — JWT claim always wins). DTOs carry no tenant_id; patientId from route. Non-spoofable. Update→404 (KeyNotFoundException) for unknown/cross-tenant id.
- **Permission model:** NEW `docslot.medical_history.create` + `.update`, both is_dangerous=t scope=tenant (excludes admin/staff auto-grants). Granted to tenant_owner (auto all-docslot block), doctor (explicit — also added the previously-missing `.read` to doctor; doctors ARE the clinical writers, correct), super_admin (universal sweep). create≠update≠read least-privilege split is correct. Live verified: only those 3 roles hold create/update; the qa_*/smoke_* test-artifact roles have read only.
- **No consent gate on WRITE:** intentional + consistent with prescription/lab WRITE handlers — writing a record is an act of care; READING is the disclosure that needs consent+purpose (the read path stays consent+purpose+break-glass gated).
