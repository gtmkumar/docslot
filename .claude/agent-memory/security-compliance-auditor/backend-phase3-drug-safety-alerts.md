## The 5 author decisions — my rulings
1. **No schema/perm change → SOUND.** drug_alerts table (03_docslot.sql:561) + RLS `tenant_isolation_drug_alerts` (05:717, FOR ALL, EXISTS-join through prescriptions.tenant_id, covers INSERT via USING-as-WITH-CHECK) pre-exist; source==bundle byte-identical (verified). drug_alerts has NO own tenant_id BY DESIGN (RLS via parent prescription join). INSERT rides the issue/amend UoW tx where parent Rx is already inserted + `app.tenant_id` is SET LOCAL (cmds are NOT ISelfManagedTransaction → UnitOfWorkBehavior wraps). No RLS hole.
2. **Plaintext drug_alerts (medication_name/description) → ACCEPTED, not required-encrypted.** Schema is authoritative + deliberately left these plaintext (idx_drug_alerts_critical partial index implies queryability; NOT in encrypted_fields_registry, unlike prescriptions.medications which IS @05:137). Author's 3 mitigations hold: (a) medication_name = just-PRESCRIBED drug (already cleartext in the prescriber's own input, appears on the Rx itself — not net-new PHI disclosure), (b) descriptions name prescribed drug + DERIVED CLASS label (AllergenLabel() returns "penicillin-class"/"sulfonamide-class"/etc — NEVER copies the patient's encrypted allergen free-text), (c) conflicting_record_id LINKS the encrypted source record instead of duplicating it. CONDITION: add a COMMENT-ON-COLUMN / registry note documenting the deliberate plaintext+class-label-only design so a future maintainer doesn't widen description to echo allergen text.
3. **Screening NOT consent-gated → ACCEPTED.** Internal SAFETY control on the prescriber's OWN create action; withholding an allergy/interaction check on a data-SHARING-consent flag would endanger the patient. Allergen text decrypted in-process only, never returned, never copied to plaintext. The READ endpoint (GET drug-alerts) IS fully consent+break-glass+purpose gated, parity with prescription read. Consistent with house rule "writing is care, reading is the gated disclosure".
4. **purpose='treatment' + notes='automated drug-safety screening' → ACCEPTED.** 'drug_safety_screening' is NOT in declared_purpose CHECK domain (05:357 — treatment/follow_up/emergency/consultation/research/audit/patient_request/legal_obligation); 'treatment' is the correct allowed value (screening IS part of treating the encounter), provenance carried in purpose_notes. NB the EncryptionContext ResourceType "drug_safety_screening" flows to key_usage_log.resource_type which is VARCHAR(50) NO CHECK → no constraint violation.
5. **Atomicity (alerts roll back with Rx) → ACCEPTED for v1.** Defensive JSON parse (malformed→empty→no alerts, never throws); curated source pure/deterministic; genuine alert-insert DB failure rolls back the Rx too (no half-issue). CONDITION (MEDIUM): the in-process PHI decrypt's purpose-of-use record is written on a DEDICATED connection (PurposeOfUseWriter uses IDedicatedConnectionFactory, commits independently — same as AuditTrailWriter), so it SURVIVES an Rx rollback = correct DPDP posture (the decrypt DID happen). BUT screening logs purpose-of-use + decrypts BEFORE EvaluateAsync; if Rx later fails you get a durable PHI-access record with no surviving alerts. Acceptable (access record should be durable), just document the asymmetry. NOT a blocker.
