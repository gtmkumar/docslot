**#12 cross-space embedding (FIXED, sound).** `ai_service/app/rag_repository.py::fetch_patient_embeddings` now also filters `AND embedding_model=%(model)s AND embedding_dimensions=%(dims)s`; `rag.py::retrieve` passes `embedder.model_label`+`embedder.dim` and keeps the `s["vector"].size == qvec.size` guard before np.vstack. Root cause real: `embeddings.py` has TWO backends sharing dim=384 but different spaces — fastembed `BAAI/bge-small-en-v1.5` vs fallback `hashing-vectorizer-384`; old `.size`-only filter let model-A query rank against model-B rows (garbage) or crash np.vstack on a cross-dim row. Tenant+patient scoping preserved (filters unchanged). No security delta — relevance/crash fix.

**#10 allows_phi egress gate (NEW, the governance control).** `ai_service/app/model_config.py::get_phi_model_config(use_case, tenant_id)` reads `ai.ai_model_configs` (schema source-of-truth, previously NEVER read). Returns a config ONLY if `is_active AND allows_phi AND bao_signed`. Resolution `(tenant_id=%(t)s::uuid OR tenant_id IS NULL) ORDER BY (tenant_id IS NOT NULL) DESC, is_default_for_use_case DESC, updated_at DESC LIMIT 1` — tenant-specific wins over platform default; a tenant can ONLY match its own row or the NULL-tenant default → no cross-tenant approval inheritance (verified). 30s `time.monotonic()` TTL cache + `reset_cache()`. `rag.py::_llm_answer` now takes tenant_id; egress only when cfg resolves, else None→LOCAL extractive (fail-CLOSED). Seeded platform default in `database/06_ai_services.sql` is allows_phi=false/bao_signed=false (correct secure default; RAG stays local until a signed BAA flips both). Bundle regenerated (docslot_complete.sql:3769).

**THE LOAD-BEARING CONDITION (Finding 1, HIGH — not a blocker because pre-existing & key-gated).** The gate covers ONLY the RAG path. `ai_service/app/triage.py::_extract_llm` (lines ~150-194) ships the patient's free-text `complaint` to the SAME external Anthropic endpoint gated ONLY by `if not api_key` — NO `get_phi_model_config`/allows_phi check. The triage router (`routers/triage.py:61-68`) ITSELF classifies the complaint as PHI when patient_id/bookingId is bound (requires X-Purpose-Of-Use + tenant + patient_tenant_links gate). So once a BAA flips allows_phi for rag_medical and someone sets AI_LLM_API_KEY, the chief-complaint PHI egresses ungoverned via triage. In 5A I wrote "triage.py touches no clinical PHI *tables*" — true for at-REST, but this slice introduces the at-EGRESS concept and makes triage the inconsistent twin. CONDITION: gate `_extract_llm` on a `triage` use_case allows_phi config (or document/justify why complaint-egress is acceptable). Mitigation today: default api_key unset AND default allows_phi=false → no live egress.
