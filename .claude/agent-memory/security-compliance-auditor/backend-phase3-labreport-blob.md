First slice to store + serve a real PHI artifact (lab-report PDF/image). NO schema change — `docslot.lab_reports` file cols (file_url/file_name/file_size_bytes/file_mime_type/uploaded_at/uploaded_by_user_id) already existed (03_docslot.sql:590-601); `file_url` now holds an OPAQUE tenant-namespaced blob storage KEY (not a URL). Build clean (only pre-existing MessagePack NU1903), targeted blob test green live.

## New seam (honest dev-adapter, NOT fake completion)
- `IBlobStorage` (mediq.Application/Abstractions/StorageAbstractions.cs): `PutAsync(tenantId,resourceType,resourceId,fileName,content,ct)->StoredBlobRef(key,size)` + `GetAsync(storageKey,tenantId,ct)->byte[]?`. CONTRACT: bytes handed in are ALREADY an encrypted envelope (app encrypts before store); adapter only holds ciphertext; every key tenant-namespaced; GetAsync refuses cross-namespace key.
- `LocalFileSystemBlobStorage` (dev default): key=`{tenantId:N}/{Sanitize(resourceType)}/{resourceId:N}/{Guid7:N}{ext}`. `InMemoryBlobStorage` (tests). DI flag `BlobStorage:Provider` (local_fs|in_memory) in DependencyInjection.cs:136-142; Singleton; prod swaps S3/GCS/Azure+SSE/KMS behind same interface (same pattern as payout rail / WhatsApp sender). appsettings: BlobStorage{Provider=local_fs,RootPath=App_Data/blobs}.

## Why PASS (all 8 gates ✅, verified source+live+empirical)
- **PHI-at-rest (genuine ciphertext):** SetLabReportFileCommand base64-decodes the upload (bad base64 -> 422 ValidationException, not 500), `EncryptBytesAsync(ClinicalFields.ReportResults, ...)` envelope-encrypts the BYTES, then `blobs.PutAsync(UTF8(envelope))`. The blob store NEVER sees plaintext. Test reads the REAL on-disk file via local_fs and asserts no "%PDF"/"Hemoglobin", and the bytes are an envelope with "keyId". Download round-trips to EXACT original bytes.
- **Encryption refactor byte-IDENTICAL (zero regression to existing encrypted fields):** `git diff main` on FieldEncryptionService.cs = pure extract-method. `EncryptAsync`=thin `EncryptCoreAsync(UTF8.GetBytes(plaintext))` (same bytes the old inline `var plainBytes=Encoding.UTF8.GetBytes` produced); `DecryptAsync`=`UTF8.GetString(DecryptCoreAsync())` (== old `return Encoding.UTF8.GetString(plain)`). Envelope format/key-resolution/GCM params/nonce(12)/tag(16)/key_usage_log ALL unchanged. New `EncryptBytesAsync`/`DecryptBytesAsync` = raw-bytes pass-through to the same core.
- **DPDP key-destruction also SEALS the file (correct + a strength):** the blob reuses `lab_reports.structured_results` FieldRef -> registry row data_class=`medical_history` (live verified, encryption_required=t). So the blob's wrapped data key is sealed under the SAME tenant `medical_history` KMS key as the structured results. CryptoErasureService destroying that key renders BOTH the structured_results column AND the blob envelope permanently unrecoverable in one erasure. Correct reuse.
- **Consent gate IDENTICAL to structured-results read:** GetLabReportFileQuery is byte-for-byte the same gate as GetLabReportQuery — require X-Purpose-Of-Use (else 422), load report (404/cross-tenant via tenant-filtered GetLabReportAsync under RLS), `patient.HasActiveConsent` OR break-glass `GetActiveGrantAsync(user,tenant,patient,"lab_report",reportId)` (specific-report grant, since a single-report download), null grant -> 403, then `purpose.RecordAsync(... isBreakGlass=grant!=null, justification)`. NO bytes streamed before the gate: blobs.GetAsync + DecryptBytesAsync run AFTER purpose-record. FileUrl null -> 404; blob missing -> 404. Test proves: consent ok->200 exact bytes, consent revoked->403, no-file->404.
- **Tenant isolation (3 layers, empirically traversal-proof):** (1) DB load filters tenant_id under RLS (docslot_app NOBYPASSRLS). (2) blob key tenant-namespaced; GetAsync string-prefix-guards `{tenantId:N}/` (fixed 32-hex from Guid:N, non-spoofable) BEFORE resolution. (3) `ResolveWithinRoot` Path.GetFullPath-normalizes then requires `full==root || full.StartsWith(root+DirSep)` — empirically blocks `../../etc/passwd`, deep `..`, and cross-tenant break-back-in (`tenantA/sub/../../../tenantB/..`); sibling-dir `blobs-evil` blocked by the trailing-sep requirement; `..%2f` stays inside tenant dir (harmless literal filename). PutAsync write path also traversal-safe: resourceType Sanitize strips non-`[A-Za-z0-9_-]`, Extension only allows `.`+1-9 alnum, both GUIDs fixed-hex, and PutAsync ALSO runs ResolveWithinRoot. r.FileUrl is itself sourced from an RLS-scoped row, so the prefix guard is genuine defense-in-depth.
- **Upload authz = WRITE, distinct perm, not consent-gated (consistent):** POST file `[docslot.report.upload]` (distinct from `docslot.report.read` on download). Upload is an act of care (mirrors UploadLabReport/IssuePrescription) — writing isn't the disclosure; READING is. Correct.
- **Audit/no-PHI:** upload_file audit row ChangeSummary=`"Lab-report file stored (encrypted; {N} bytes)"` (N=PLAINTEXT size; benign metadata, no content). key_usage_log = key_id/op/actor only. Storage key = GUIDs only, no PHI. Download path records purpose-of-use (no PHI), no audit "read" event (same as other clinical reads).
- **Idempotency leak (Form 16A lesson) N/A:** download is an `IQuery` -> command-only IdempotencyBehavior never touches it; decrypted bytes never hit the plaintext idempotency store. SetLabReportFileResult=(ReportId,SizeBytes) only — even if the upload command is cached with an Idempotency-Key, the response carries no PHI and the request body (the base64 PHI) is NOT cached. No IDoNotCacheResponse needed.

## CONDITIONS / follow-ups (pre-prod, none blocking)
1. (MEDIUM) ~~No upload size bound on base64-in-JSON.~~ **CLOSED in-PR (re-verified live 2026-06-29).** Two layers: (a) SetLabReportFileValidator `MaxBase64Length=28_000_000` (~20MB file) `.Must(len<=cap)` -> 422 in the FluentValidation behavior BEFORE the handler decodes/copies; (b) ClinicalController `[RequestSizeLimit(30_000_000)]` on the POST file action ONLY -> 413 at the pipeline BEFORE model binding buffers the JSON. Pipeline limit (30M) > validator cap (28M) so in-band-too-large gets the friendly 422 and anything beyond gets 413. Attribute action-scoped (doesn't loosen other endpoints); length check on already-bound string (no PHI logged); encryption/consent/tenant/audit paths untouched. Build 0 errors, ClinicalPhiTests 10/10 (small-PDF test still passes under cap).
2. (LOW) **Blob is not deleted on DPDP erasure beyond key destruction.** Erasure seals it cryptographically (unrecoverable) but the ciphertext file lingers on disk/object store. Acceptable per ADR-005 (crypto-erasure = key destruction, not row/blob delete) but add object lifecycle/GC so dead ciphertext is eventually purged.
3. (INFO) file_size_bytes = PLAINTEXT size (envelope is larger) — acceptable/intended (UI wants real size), not PHI.
4. (INFO) local_fs disk holds ONLY the envelope (confirmed by test) — the dev adapter is legitimate behind the flag; prod = object store + provider SSE/KMS as a SECOND at-rest layer under the app envelope.
