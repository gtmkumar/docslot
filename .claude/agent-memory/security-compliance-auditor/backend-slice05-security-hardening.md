**GATE 7 — AUDIT-CHAIN CONCURRENCY RACE (tracked pre-GA fix, NOT a hard blocker now):** `append_to_audit_chain` (`05_security_hardening.sql:176`) reads chain head (`SELECT row_hash ORDER BY sequence_number DESC LIMIT 1`) then inserts, with NO advisory lock / serialization (grep: 0 pg_advisory in file). Two concurrent audit_log INSERTs both read the same previous_hash → fork → verify_audit_chain reports a break. sequence_number BIGSERIAL UNIQUE gives distinct seq but does NOT serialize the head-read. Under real concurrent load = false tamper alarms (or a window masking real tampering). Why tracked not blocker: no PHI served this slice, audit volume is low/serial in current flows, chain currently intact (hand-repaired, holds). MUST fix before GA / concurrent load. Fix: `pg_advisory_xact_lock(<audit-chain-key>)` at the top of append_to_audit_chain (serializes appenders), OR a single-writer appender, OR per-tenant chains.

**GATE 1 + 8 — GUC TRUST MODEL (prod-hardening, confine before clinical PHI):** The append-only escape hatch `app.allow_audit_maintenance='on'` AND the RLS bypass `app.is_super_admin='true'` are custom session GUCs the APP DB ROLE CAN FREELY SET (verified: `SET app.allow_audit_maintenance='on'` and `SET app.is_super_admin='true'` both succeed as the app role). This is inherent to Postgres session-GUC RLS — the standard mitigation is a NON-SUPERUSER, NON-BYPASSRLS app role that the app trusts to set GUCs honestly from validated claims (app.tenant_id is set correctly via SET LOCAL). CURRENT DEV DB: connecting role `gtmkumar` is `rolsuper=t, rolbypassrls=t` → RLS is ENTIRELY BYPASSED today (superuser ignores all policies). Acceptable now (no PHI served), but BEFORE 03b clinical PHI: (a) run app under a dedicated non-super/non-BYPASSRLS role; (b) the audit-maintenance hatch should be settable only by a privileged retention role (app code never sets it — verified — but the role CAN); consider ALTER-restricting or moving maintenance to a separate role. The hatch itself is a reasonable design IF confined; flag as confinement-required.
